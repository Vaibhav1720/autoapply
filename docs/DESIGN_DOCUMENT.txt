A.1 PROBLEM STATEMENT
---------------------
Job seekers spend hours manually applying to jobs on multiple portals.
AutoApply automates this by collecting user data once and auto-applying to
selected companies/roles via API integration, email, or browser automation.

A.2 TECH STACK
--------------
- Frontend:      Flutter (Android + Web from single codebase)
- Backend:       Azure Functions (Python) — serverless, cost-efficient
- Database:      Azure Cosmos DB (NoSQL — flexible profile/job schemas)
- File Storage:  Azure Blob Storage (resumes, cover letters, documents)
- Auth:          Azure AD B2C (social + email login)
- AI:            Azure OpenAI (cover letter generation, job matching)
- Search:        Cosmos DB vector search (DiskANN) + full-text search
                 (upgrade to Azure AI Search only if scale demands)
- Queue:         Azure Storage Queues (Phase 5) → Azure Service Bus (Phase 7)
- Notifications: Azure Notification Hubs (push) + Azure Communication Services (email)
- Job Ingestion: Azure Functions (timer-triggered scrapers)
- Cache:         Azure Redis Cache (deferred to Phase 7)
- CI/CD:         Azure DevOps Pipelines
- Gateway:       Azure Functions direct HTTPS (Phases 2-7) → APIM Consumption (Phase 8)

BUDGET STRATEGY ($100/month cap):
  Phase 1: $0       (local only)
  Phase 2-3: ~$2    (Cosmos DB Free Tier, Blob, B2C Free)
  Phase 4: ~$15-25  (Azure OpenAI pay-per-token, Document Intelligence)
  Phase 5-6: ~$25-40 (Storage Queues, ACI on-demand, Communication Services)
  Phase 7: ~$55-75  (Redis $16, Service Bus $10, Notification Hubs Free)
  Phase 8: ~$75-100 (APIM Consumption ~$3.50/M calls)

B.1.4 DATA LAYER ARCHITECTURE
-------------------------------

Azure Cosmos DB (SQL API, Multi-region):
  +---------------------------+  +---------------------------+
  | Container: profiles       |  | Container: companies      |
  | Partition: /userId        |  | Partition: /industry      |
  |                           |  |                           |
  | - UserProfile             |  | - Company                 |
  | - UserPreferences         |  | - CompanyStats            |
  | - OnboardingState         |  |                           |
  +---------------------------+  +---------------------------+

  +---------------------------+  +---------------------------+
  | Container: jobs           |  | Container: applications   |
  | Partition: /companyId     |  | Partition: /userId        |
  | Vector Index: DiskANN     |  |                           |
  |   (embedding, cosine,     |  | - Application             |
  |    1536 dims)             |  | - StatusHistory           |
  | - Job (with embedding)    |  | - AutoApplyLog            |
  +---------------------------+  +---------------------------+

  NOTE on /companyId partition: Acceptable for early scale. If large
  companies (Microsoft, Google) cause hot partitions at volume, migrate
  to synthetic key: {companyId}-{hash(id) % 10}. Re-evaluate at Phase 7.

Azure Blob Storage:
  resumes/{userId}/          - resume_v1.pdf, resume_v2.pdf
  cover-letters/{userId}/    - {jobId}.pdf, {jobId}.docx
  company-logos/{companyId}/ - logo.png

Cosmos DB Vector Search (replaces Azure AI Search — saves $75/month):
  jobs container:
    Vector index policy: DiskANN, cosine similarity, 1536 dimensions
    Full-text search: Cosmos DB CONTAINS + ORDER BY scoring
    Vector search: VectorDistance() function for KNN queries
    Filters: isActive, expiresAt, location, jobType, companyId
  companies container:
    Full-text search: CONTAINS on name, industry
    Filters: industry, location, size, rating

Azure Redis Cache:
  Key Patterns:
    company:list:{industry}:{page}    -> 5 min TTL
    company:detail:{companyId}        -> 15 min TTL
    jobs:trending                     -> 10 min TTL
    user:matchcache:{userId}:{hash}   -> 30 min TTL
    user:applylimit:{userId}:{date}   -> 24h TTL (daily apply counter)


B.1.5 AUTHENTICATION & SECURITY FLOW
---------------------------------------

  Flutter App ---> Azure AD B2C ---> Identity Providers
                   - Sign-up          - Google
                   - Sign-in          - LinkedIn
                   - Password Reset   - Email/Pass
       |
       | JWT (id_token + access_token)
       v
  Azure Functions (Direct HTTPS — Phases 2-7)
    Auth Middleware (python-jose):
    - Validates JWT signature against B2C JWKS endpoint
    - Checks audience (client_id)
    - Checks issuer (B2C tenant URL)
    - Extracts userId from 'sub' claim (NO X-User-Id header)
    - Enforces resource ownership
    - Role-based access (user/admin)
    - In-memory rate limiting per userId
    Data Isolation:
    - All queries scoped to userId
    - Cosmos DB partition = userId
    - Blob paths = /{userId}/...

  Phase 8 Addition (APIM Consumption tier):
    - APIM added as gateway in front of Functions
    - validate-jwt policy (defense-in-depth, second layer)
    - Rate limiting (100 req/min per user)
    - Response caching (5 min TTL)
    - CORS policy
    - Functions middleware KEPT as defense-in-depth

B.2.3 STATE MANAGEMENT ARCHITECTURE (Provider)
-------------------------------------------------

  MaterialApp
    -> MultiProvider
       +- ChangeNotifierProvider<AuthProvider>
       +- ChangeNotifierProvider<ProfileProvider>
       +- ChangeNotifierProvider<CompanyProvider>
       +- ChangeNotifierProvider<JobProvider>
       +- ChangeNotifierProvider<ApplicationProvider>
          -> Router
             -> Screens (consume via context.watch/read)

  AuthProvider:
    State: user, token, isLoggedIn
    Actions: login(), logout(), refreshToken()
    Depends on: AuthService, SecureStorage

  ProfileProvider:
    State: profile, isOnboarded, loading, currentStep
    Actions: loadProfile(), updateProfile(), uploadResume(), parseResume(),
             nextStep(), prevStep()

  CompanyProvider:
    State: companies, selected, filters, following, loading
    Actions: loadCompanies(), searchCompanies(), follow(id), applyFilters()

  JobProvider:
    State: jobs, matched, filters, selected, loading
    Actions: loadJobs(), searchJobs(), getMatches(), applyFilters()

  ApplicationProvider:
    State: applications, queue, stats, chartData, selectedApp, loading
    Actions: loadApplications(), loadStats(), applyToJob(jobId),
             autoApply(List<jobId>), cancelQueued(appId), updateNotes()

B.2.7 OFFLINE & ERROR HANDLING STRATEGY
------------------------------------------

  Online Mode:
    - Normal API calls

  Offline Mode:
    - Show cached data from Hive local DB
    - Queue write operations locally
    - Show offline banner
    - Sync on reconnect

  Error Type -> UI Handling:
    401 Unauthorized   -> Redirect to login
    429 Rate Limited   -> Show "try again" toast
    500 Server Error   -> Retry + error screen
    Network Error      -> Offline mode
    Validation Error   -> Inline field errors

C.5 AUTO-APPLY QUEUE PROCESSING — DETAILED ALGORITHM
-------------------------------------------------------

Function: process_application (Queue Triggered)
Trigger: Azure Storage Queue "auto-apply-queue" (Phase 5-6)
         Azure Service Bus Queue "auto-apply-queue" (Phase 7+)

Algorithm:

  1. RECEIVE message from Service Bus queue
     Input: { userId, jobId, companyId, applicationId, applyMethod, priority }

  2. UPDATE application status in Cosmos DB
     status: "queued" -> "processing"
     Add statusHistory entry

  3. LOAD user profile from Cosmos DB
     Get: personal info, skills, experience, resume URL

  4. LOAD job details from Cosmos DB
     Get: title, description, requirements, company info

  5. CHECK if auto-generate cover letter is enabled
     IF enabled:
       a. Call generate_cover_letter(userProfile, jobDetails)
       b. Upload generated PDF to Blob: cover-letters/{userId}/{jobId}.pdf
       c. Get cover letter URL
     ELSE:
       a. Use user's default cover letter template
       b. OR skip cover letter

  6. DETERMINE apply method based on job.applyMethod:

     CASE "api":
       a. Load company API configuration
       b. Build request payload (map profile fields to API format)
       c. POST to company API endpoint
       d. Parse response for confirmation

     CASE "email":
       a. Build email body (cover letter + intro)
       b. Attach resume PDF
       c. Send via Azure Communication Services
       d. Log email message ID

     CASE "browser":
       a. Spin up Azure Container Instance with Playwright
       b. Navigate to application URL
       c. Fill form fields using profile data mapping:
          - Name fields <- personal.firstName, personal.lastName
          - Email <- email
          - Phone <- personal.phone
          - LinkedIn <- personal.linkedinUrl
          - Resume upload <- download & re-upload
          - Cover letter text <- generated or template
          - Custom questions <- AI-generated answers
       d. Submit form
       e. Capture screenshot for verification
       f. Tear down container

  0. IDEMPOTENCY GUARD (before step 1):
     Check application status in Cosmos — if already "applied" or "failed",
     skip processing (prevents duplicate applications on Queue redelivery)

  7. HANDLE result:
     IF success:
       a. Update Cosmos: status = "applied", appliedAt = now
       b. Send push notification: "Applied to {jobTitle} at {companyName}"
          (basic log in Phase 5-6; push notification in Phase 7)
       c. Increment daily counter (Cosmos doc Phase 5-6; Redis Phase 7+)
     IF failure:
       a. Log error details
       b. IF attempts < 3:
          - Phase 5-6: Re-enqueue to Storage Queue with visibility
            timeout (backoff: 5min, 15min, 60min)
          - Phase 7+: Requeue to Service Bus with scheduled delivery
          - Update Cosmos: attempt count
       c. IF attempts >= 3:
          - Update Cosmos: status = "failed", lastError = errorMessage
          - Phase 5-6: Failed status tracked in Cosmos autoApplyLog
            (no DLQ — Storage Queues don't support it)
          - Phase 7+: Message moves to Service Bus dead-letter queue
          - Send notification: "Failed to apply to {jobTitle}"
       d. Log to Application Insights


C.6 AI MATCHING — DETAILED ALGORITHM
---------------------------------------

Function: match_jobs(userId)

Algorithm:

  1. LOAD user profile from Cosmos DB
     Required: skills, experience, preferences, profileEmbedding

  2. CHECK Redis cache
     Key: user:matchcache:{userId}:{hash(preferences)}
     IF cache hit AND age < 30 min: return cached results

  3. GENERATE profile embedding (if not exists or stale)
     Input text: concatenate(skills, latest_job_title, experience_summary, preferences)
     Model: Azure OpenAI text-embedding-3-large (1536 dimensions)
     Store embedding in Cosmos for reuse

  4. VECTOR SEARCH via Cosmos DB (integrated DiskANN vector search)
     Container: jobs
     Query: VectorDistance(embedding, profileEmbedding, 'cosine'), TOP 100
     Filters:
       - isActive = true
       - expiresAt > now
       - location IN user.preferences.locations OR remote = true
       - jobType IN user.preferences.jobType

  5. CALCULATE detailed match scores for top 100 results:

     For each job:
       a. skills_score = |intersection(user.skills, job.skills)| / |job.skills| * 100
       b. experience_score = calculate_exp_match(user.experience, job.requirements)
          - Years match: user_years >= job.min_years ? 100 : (user_years/min_years)*100
          - Relevant title match: fuzzy_match(user.titles, job.title) * weight
       c. location_score:
          - Exact city match: 100
          - Same state: 80
          - Remote and user wants remote: 100
          - Different location but willing to relocate: 60
          - No match: 0
       d. salary_score:
          - Job range overlaps user range: 100
          - Partial overlap: proportional
          - No overlap: 0 (exclude from results if > 30% gap)
       e. overall_score = (skills_score * 0.35) + (experience_score * 0.30)
                        + (location_score * 0.20) + (salary_score * 0.15)

     BOOST factors:
       - Job from followed company: +5 points
       - Job from preferred company: +3 points
       - Recently posted (< 3 days): +2 points

  6. SORT by overall_score descending

  7. CACHE result in Redis (30 min TTL)

  8. RETURN top N results with full match breakdown

Retry Strategy:
  - Cosmos DB: SDK built-in retry (429s)
  - Blob Storage: 3 retries, exponential backoff
  - Azure OpenAI: 3 retries, 429 rate limit with backoff (2s, 4s, 8s)
  - Service Bus: Message lock renewal, dead-letter after 5 failures
  - Browser automation: 3 attempts per application

Circuit Breaker (per external service):
  - Track failure count per service in Redis
  - If > 10 failures in 5 minutes: OPEN circuit
  - Wait 60 seconds, then HALF-OPEN (allow 1 request)
  - If success: CLOSE circuit; if failure: OPEN again


C.10 CACHING STRATEGY (Redis)
-------------------------------

Cache Tiers:
  L1 - In-memory (Flutter): Hive/SharedPreferences (user profile, settings)
  L2 - Redis (Azure): Frequently accessed data
  L3 - Cosmos DB: Source of truth

Redis Patterns:
  Read-through cache:
    1. Check Redis for key
    2. If miss: query Cosmos DB, store in Redis with TTL
    3. Return data

  Write-through invalidation:
    1. Write to Cosmos DB
    2. Delete related Redis keys
    3. Next read populates cache

  Key        | Pattern              | TTL     | Purpose
  -----------|----------------------|---------|---------------------------
  company:*  | Read-through         | 5-15min | Company lists/details
  jobs:trend | Read-through         | 10min   | Trending jobs
  user:match | Read-through         | 30min   | Match results (per user)
  user:limit | Increment + expire   | 24h     | Daily apply limit counter
  session:*  | Set + expire         | 1h      | Active session data

     - App Settings: Cosmos connection (Key Vault refs), OpenAI key, etc.
     - Slots: staging + production
     - CORS: Configured on Functions host (no APIM until Phase 8)
     - Auth: JWT validation in middleware (python-jose)

  4. Azure API Management (Phase 8 only)
     - Tier: Consumption (~$3.50 per million API calls)
     - Policies: JWT validation, rate limiting, CORS, versioning

  5. Azure Blob Storage Account

     - Containers: resumes, cover-letters, company-logos

     - Lifecycle: Move to cool tier after 90 days

  6. Azure OpenAI (Phase 4+)
     - Model deployments: gpt-4o, text-embedding-3-large
     - TPM limits: 60K tokens/min

  7. Azure Document Intelligence (Phase 4+)
     - Tier: Pay-as-you-go (~$1.50/1000 pages)
     - Model: prebuilt-document (resume OCR)

  8. Azure Service Bus (Phase 7+ only)
     - Tier: Standard (~$10/month)
     - Queue: auto-apply-queue (with dead-letter)
     - TTL: 24 hours

  9. Azure Redis Cache (Phase 7+ only)
     - Tier: Basic C0 (~$16/month)
     - Eviction: allkeys-lru

  10. Azure Notification Hubs (Phase 7+)
      - Tier: Free (1M pushes/month)
      - Platforms: FCM (Android), Web Push

  11. Azure Communication Services (Phase 5+)
      - Email domain configured
      - Pay-per-email: ~$0.25/1000 emails

  12. Azure Container Instances (Phase 5+, on-demand)
      - Image: custom Playwright image
      - CPU: 1 core, Memory: 2GB
      - Cost: ~$0.05/hour (~$0.002 per application at ~2 min)
      - Lifecycle: Create per job, destroy after completion

  13. Application Insights
      - Tier: Free (5 GB/month included)
      - Connected to Functions
      - Log Analytics workspace

  14. Azure Static Web Apps (Flutter web hosting)
      - Tier: Free

  NOT provisioned (cost savings):
    - Azure AI Search: Replaced by Cosmos DB vector search (saves ~$75/month)
    - APIM in Phases 2-7: Functions direct HTTPS (saves ~$49/month)
    - Service Bus in Phases 5-6: Storage Queues used (saves ~$10/month)
    - Redis in Phases 1-6: Direct Cosmos DB queries (saves ~$16/month)

PHASE 4 -- AI Features (~$15-25/month)
------------------------------------------------------------------------
Goal: AI-powered job matching, semantic search, resume parsing, and
cover letter generation are functional.

CHANGED FROM ORIGINAL: Use Cosmos DB integrated vector search instead of
Azure AI Search. Saves $75/month.

  Backend:
    1. Set up Azure OpenAI client (shared/openai_client.py)
       - GPT-4o for text generation (~$5/1M input, ~$15/1M output)
       - text-embedding-3-large for embeddings (~$0.13/1M tokens)
    2. Enable vector search on Cosmos DB jobs container:
       - Add vector index policy (DiskANN, cosine, 1536 dimensions)
       - Add embedding vector field to job documents
       - NO separate Azure AI Search service needed
    3. POST /api/v1/profile/resume/parse -- AI resume parsing flow:
       Azure Document Intelligence OCR -> OpenAI structured extraction
       (~$1.50/1000 pages)
    4. GET /api/v1/jobs/match -- AI job matching:
       a. Generate/retrieve user profile embedding
       b. Vector search via Cosmos DB VectorDistance() function (top 100)
       c. Calculate detailed match scores (skills 35%, experience 30%,
          location 20%, salary 15%)
       d. Apply boost factors (followed company +5, recent +2)
       e. Return sorted results with match breakdowns
       NOTE: No cache until Phase 7 -- results computed on each request
    5. Upgrade POST /api/v1/jobs/search -- hybrid search:
       Cosmos DB full-text search + vector search (both built-in)
    6. Upgrade POST /api/v1/companies/search -- Cosmos DB full-text search
    7. Build cover letter generator (generate_cover_letter.py):
       GPT-4o with detailed prompt -> PDF via reportlab -> Blob upload
    8. Update profile create/update to generate profile embeddings
    9. Build job ingestion scrapers (timer-triggered, every 6h):
       scrape_linkedin.py, scrape_indeed.py, scrape_glassdoor.py,
       normalize_jobs.py -- normalize, embed, write to Cosmos with vectors
    10. Unit + integration tests

  Frontend:
    1. Update JobProvider with getMatches() for AI-matched jobs
    2. MatchScoreIndicator -- detailed breakdown (skills, exp, loc, salary)
    3. Update JobFeedScreen -- "For You" tab showing AI matches
    4. Update JobDetailScreen -- match score breakdown display
    5. Resume parse flow in onboarding: upload -> parse -> review ->
       confirm extracted data
    6. Cover letter preview in job detail screen

  Cost breakdown:
    - Azure OpenAI: ~$10-20/month (moderate dev usage)
    - Document Intelligence: ~$1-2/month (sporadic resume parsing)
    - Cosmos DB vector search: included in RU cost (no extra charge)

  Deliverable: Users see AI-matched jobs with scores, can search with
  semantic understanding, parse resumes via AI, and preview cover letters.

PHASE 5 -- Auto-Apply Engine (~$25-40/month)
------------------------------------------------------------------------
Goal: Users can queue jobs for automatic application via API, email, or
browser automation.

CHANGED FROM ORIGINAL: Use Azure Storage Queues (same Storage Account as
Blob -- ~$0.01/month) instead of Service Bus ($10/month).

  Backend:
    1. Set up Azure Storage Queue client (shared/queue_client.py)
       - Queue: auto-apply-queue (uses same Storage Account as Blob)
       - Note: No dead-letter support. Failed messages tracked in Cosmos
    2. Set up applications container in Cosmos DB
    3. POST /api/v1/applications -- manual apply to a job
    4. GET  /api/v1/applications -- list user applications
    5. GET  /api/v1/applications/{id} -- application detail
    6. PUT  /api/v1/applications/{id}/status -- update status
    7. POST /api/v1/auto-apply -- queue jobs for auto-apply:
       a. Max 20 jobIds per request
       b. Daily limit check (Cosmos DB counter doc per userId+date)
       c. Profile completeness check
       d. Duplicate application check
       e. Blacklisted company check
       f. Create Application records (status: queued)
       g. Send messages to Storage Queue
       h. Increment daily counter in Cosmos
    8. GET  /api/v1/auto-apply/queue -- view pending queue
    9. DELETE /api/v1/auto-apply/queue/{id} -- cancel queued application
    10. PUT /api/v1/auto-apply/settings -- update auto-apply preferences
    11. process_application.py (Queue-triggered Azure Function):
        a. Idempotency guard: check application status before processing
        b. Load user profile + job details
        c. Generate cover letter if enabled (Phase 4 function)
        d. Apply via method:
           - "api": POST to company API endpoint
           - "email": Send via Azure Communication Services (~$0.25/1000)
           - "browser": Spin up ACI with Playwright, fill form, submit
             (~$0.002 per application at ~2 min)
        e. Update status in Cosmos
        f. Retry on failure (max 3, visibility timeout: 5min, 15min, 60min)
        g. After 3 failures -> status = "failed" in Cosmos autoApplyLog
    12. Unit + integration tests for queue processing

  Frontend:
    1. ApplicationProvider -- loadApplications(), applyToJob(),
       autoApply(jobIds), cancelQueued(), loadQueue()
    2. ApplicationService -- HTTP calls
    3. Update JobDetailScreen -- "Apply" and "Auto-Apply" buttons wired
    4. ApplicationListScreen -- filter tabs (All, Queue, Applied, Review,
       Offer, Failed)
    5. ApplicationDetailScreen -- status timeline, company info, cover
       letter link, notes
    6. Auto-Apply Settings Screen -- max per day, blacklist management,
       cover letter toggle, review before send toggle

  Cost breakdown:
    - Storage Queue: ~$0.01/month
    - ACI: ~$0.05/hour, ~$0.002/application (2-min run)
    - Communication Services: ~$0.25/1000 emails
    - OpenAI (cover letters): included in Phase 4 estimate

  Deliverable: Full auto-apply pipeline works. Users can queue jobs,
  applications are processed asynchronously, and status is tracked.

PHASE 7 -- Scale & Production Hardening (~$55-75/month)
------------------------------------------------------------------------
Goal: Performance optimization with Redis caching, queue upgrade to
Service Bus, push/email notifications, offline support.

  Backend -- Redis Caching:
    1. Provision Azure Redis Cache Basic C0 ($16/month)
    2. Set up Redis client (shared/redis_client.py)
    3. Implement read-through cache pattern:
       - company:list:{industry}:{page}     -> 5 min TTL
       - company:detail:{companyId}         -> 15 min TTL
       - jobs:trending                      -> 10 min TTL
       - user:matchcache:{userId}:{hash}    -> 30 min TTL
    4. Migrate daily apply limit counter from Cosmos DB document to
       Redis: user:applylimit:{userId}:{date} -> 24h TTL
    5. Implement write-through invalidation on profile/company updates
    6. Circuit breaker pattern per external service:
       Track failure counts in Redis, open after 10 failures in 5 min,
       half-open after 60s cooldown

  Backend -- Queue Upgrade:
    7. Provision Azure Service Bus Standard ($10/month)
    8. Migrate from Storage Queue to Service Bus:
       - Add dead-letter queue for failed applications
       - Add message sessions for ordered processing per user
       - Update process_application.py trigger from Queue to Service Bus
       - Update shared/service_bus_client.py
    9. Backfill any failed items from Cosmos autoApplyLog

  Backend -- Notifications:
    10. Set up Azure Notification Hubs Free tier (1M push/month)
    11. Set up Azure Communication Services (already provisioned Phase 5)
    12. Build notification_service.py -- push + email
    13. Trigger notifications on: application status change, auto-apply
        completion/failure, new high-match jobs (daily digest)

  Frontend -- Notifications & Offline:
    1. Push notification registration + handling
    2. NotificationService -- FCM integration (via Azure Notification Hubs)
    3. Offline mode with Hive local cache:
       - Cache profile, recent companies, recent jobs locally
       - Queue write operations when offline
       - Show OfflineBanner widget
       - Sync on reconnect
    4. Pull-to-refresh on all list screens
    5. Loading shimmer animations during data fetch
    6. Error handling UX: 401 -> login, 429 -> toast, 500 -> retry

  Cost breakdown:
    - Redis Basic C0: $16/month
    - Service Bus Standard: $10/month
    - Notification Hubs Free: $0

  Deliverable: App is significantly faster with cached reads. Users
  receive push/email notifications. App works offline with cached data.
  Queue processing has proper dead-letter support.

SERVICE DEFERRAL NOTES
================================================================================

Until Phase 7, the following substitutions apply:

  Feature                | Without (Phases 1-6)            | With (Phase 7+)
  -----------------------|---------------------------------|---------------------------
  API Gateway            | Functions direct HTTPS          | APIM Consumption (Phase 8)
  Search/Matching        | Cosmos DB VectorDistance()      | Same (upgrade to AI Search
                         | + CONTAINS full-text            | only if perf insufficient)
  Queue                  | Azure Storage Queue             | Azure Service Bus (Phase 7)
  Dead-letter queue      | Failed status in Cosmos         | Service Bus DLQ (Phase 7)
  Company/job lists      | Direct Cosmos DB query          | Redis read-through cache
  Match results cache    | Re-compute on each request      | 30 min TTL cache
  Daily apply limit      | Cosmos DB counter doc per       | Redis INCR + 24h EXPIRE
                         | userId+date (query on apply)    |
  Trending jobs          | Cosmos DB query                 | 10 min TTL cache
  Push notifications     | Console log / in-app only       | Azure Notification Hubs
  Circuit breaker        | Basic in-memory counter         | Redis-backed counters

  Performance impact: Expect ~50-200ms added latency on cacheable reads.
  Acceptable for development and early users.

  Cost impact: Phases 1-6 total ~$25-40/month vs ~$200+/month if all
  services were provisioned from the start.
